# Firmware Update v2 — Deploy & Rollback Runbook

**Branch:** `65-franchisee-ownership-module` (carries migration 35)
**Last updated:** 2026-05-15
**Owner:** whoever is on deploy duty

This runbook covers the deploy of the firmware-update redesign:

- Migration 35 — collapses `FirmwareUpdateStatusEnum` (8 states → 4: PENDING/INSTALLED/FAILED/CANCELLED), renames `retry_count → attempt_count`, adds `last_attempt_at` + `next_retry_at` for the CSMS-side retry scheduler, adds `firmware_file.s3_key`. Migration was generated by `aerich migrate` (not hand-written).
- Rewrite of `services/firmware_update_service.py` — BootNotification-driven completion + exponential backoff (5min → 30min → 2h → 4h, capped at 5 attempts / 6h).
- Demotion of `FirmwareStatusNotification` handler to logging-only (Quectel modem WS drops during firmware download make FSN unreliable).
- S3 storage for firmware binaries (presigned URLs that span the retry window).
- New admin endpoints: `mark-installed`, `mark-failed` for polling-only chargers.
- New polling-endpoint query param: `current_firmware_version` auto-closes a PENDING update when the charger reports the target version is already live.
- Frontend admin page surfacing `attempt_count`, `last_attempt_at`, `next_retry_at` + manual close-out buttons.

This deploy is **lower risk than GST**:
- Migration 35 doesn't rewrite any business data (no `invoice_number` mutation, no row deletion).
- The downgrade SQL is clean (`ALTER COLUMN ... RENAME`, `DROP COLUMN`).
- Legacy enum values on both `firmware_update` databases were **already cleaned to FAILED on 2026-05-15** before deploy — see the [pre-deploy cleanup verification](#3-pre-deploy-cleanup-verification) section.

But it's not zero-risk. Read sections 0, 1, and the rollback section before running anything.

## 0. One-time S3 prerequisites

Firmware binaries are uploaded to S3 on upload, then served via presigned GET URLs to chargers. Existing `FirmwareFile` rows with `s3_key=NULL` still resolve via the legacy `/firmware/<filename>` StaticFiles mount, so deploying without backfilling is safe. But **any new firmware upload after deploy will fail at runtime** if the bucket isn't provisioned.

Do this once per environment, before deploy.

```bash
# From your laptop (or any machine with AWS creds)
export AWS_PROFILE=voltlync
export REGION=ap-south-1
export BUCKET=voltlync-firmware-staging          # voltlync-firmware-prod for prod

# Create the bucket, block public access, AES256 encryption
aws s3api create-bucket --bucket $BUCKET --region $REGION \
  --create-bucket-configuration LocationConstraint=$REGION
aws s3api put-public-access-block --bucket $BUCKET \
  --public-access-block-configuration "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"
aws s3api put-bucket-encryption --bucket $BUCKET \
  --server-side-encryption-configuration '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}'

# CORS — NOT strictly required for charger downloads (chargers fetch the
# presigned URL directly from their modem, not a browser). But if the admin
# UI ever adds a "download from S3" link, browsers will need this.
aws s3api put-bucket-cors --bucket $BUCKET --cors-configuration '{
  "CORSRules": [{
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["GET", "HEAD"],
    "AllowedOrigins": [
      "https://staging.voltlync.com",
      "https://app.voltlync.com",
      "http://localhost:3000"
    ],
    "ExposeHeaders": ["ETag", "Content-Length"],
    "MaxAgeSeconds": 3600
  }]
}'

# Find the EC2 instance role for the staging instance
aws ec2 describe-instances --instance-ids i-00fd9fb3c2b48932a \
  --query "Reservations[0].Instances[0].IamInstanceProfile.Arn" --output text
# Note the role name from the end of the ARN

# Attach an inline policy granting S3 access on this bucket only
aws iam put-role-policy --role-name <ROLE_NAME> \
  --policy-name VoltlyncFirmwareS3Access \
  --policy-document file://<(cat <<EOF
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::$BUCKET",
      "arn:aws:s3:::$BUCKET/*"
    ]
  }]
}
EOF
)
```

Verify from inside the backend container post-deploy:

```bash
docker exec ocpp-backend-staging python -c \
  "from services import storage_service; print(storage_service._firmware_bucket())"
# Expect: voltlync-firmware-staging  (NOT a RuntimeError)
```

For prod, instance ID is `i-0df24c96c4d5e890a`, bucket is `voltlync-firmware-prod`.

## 3. Pre-deploy cleanup verification

**This was already done on 2026-05-15.** Re-verify that no rows in `firmware_update` have the legacy enum values before deploying — the new code's `CharEnumField` only accepts `PENDING`/`INSTALLED`/`FAILED`/`CANCELLED`. Any row with `DOWNLOADING`/`DOWNLOADED`/`INSTALLING`/`DOWNLOAD_FAILED`/`INSTALLATION_FAILED` will raise `ValueError` when read.

```bash
# On the EC2
docker exec ocpp-postgres-staging psql -U ocpp_staging -d ocpp_staging_db -c \
  "SELECT status, COUNT(*) FROM firmware_update GROUP BY status ORDER BY status;"
```

Expected output (post-cleanup):

```
   status   | count
------------+-------
 CANCELLED  |     1
 FAILED     |     3
 INSTALLED  |     3
```

If you see ANY of `DOWNLOADING`, `DOWNLOADED`, `INSTALLING`, `DOWNLOAD_FAILED`, `INSTALLATION_FAILED` — **STOP** and run the cleanup SQL below before deploying:

```sql
BEGIN;
UPDATE firmware_update
   SET status = 'FAILED',
       completed_at = COALESCE(completed_at, NOW()),
       error_message = COALESCE(error_message, '') || ' [legacy state migrated to FAILED on v2 deploy]'
 WHERE status IN ('DOWNLOADING','DOWNLOADED','INSTALLING');
UPDATE firmware_update
   SET status = 'FAILED'
 WHERE status IN ('DOWNLOAD_FAILED','INSTALLATION_FAILED');
COMMIT;
```

Prod expected post-cleanup:

```
   status   | count
------------+-------
 CANCELLED  |     1
 FAILED     |     8
 INSTALLED  |     2
 PENDING    |     1
```

(The PENDING row is a normally-pending update that was scheduled but never attempted — safe to leave; the new scheduler will pick it up post-deploy.)

```bash
# From your laptop
make staging-push                       # force-push HEAD → origin/develop

# SSM in
make staging-ssm
cd /home/ec2-user/ocpp-server

# Pre-flight (see sections 2 + 3)

  "SELECT status, COUNT(*) FROM firmware_update GROUP BY status ORDER BY status;"
# Confirm no DOWNLOADING/DOWNLOADED/INSTALLING/DOWNLOAD_FAILED/INSTALLATION_FAILED rows

# Add bucket env var if not already present
nano .env.staging
# Add: AWS_S3_FIRMWARE_BUCKET=voltlync-firmware-staging

```

Only one script runs: `aerich upgrade` via `make staging-migrate`. No follow-up data backfill is needed because we already cleaned legacy enum values out-of-band.

## 5. Post-deploy verification

### 5a. Schema sanity

```bash
docker exec ocpp-postgres-staging psql -U ocpp_staging -d ocpp_staging_db -c \
  "SELECT column_name FROM information_schema.columns
    WHERE table_name='firmware_update'
      AND column_name IN ('attempt_count','last_attempt_at','next_retry_at','retry_count')
    ORDER BY column_name;"
```

Expected: `attempt_count`, `last_attempt_at`, `next_retry_at` (no `retry_count`).

```bash
docker exec ocpp-postgres-staging psql -U ocpp_staging -d ocpp_staging_db -c \
  "SELECT column_name FROM information_schema.columns
    WHERE table_name='firmware_file' AND column_name='s3_key';"
```

Expected: one row, `s3_key`.

### 5b. S3 wired up

```bash
docker exec ocpp-backend-staging python -c \
  "from services import storage_service; print(storage_service._firmware_bucket())"
# Expect: voltlync-firmware-staging  (NOT a RuntimeError)
```

### 5c. Scheduler started

```bash
docker logs ocpp-backend-staging 2>&1 | grep -i "firmware update service" | tail -3
```

Expected line:

```
📦 ✅ Started firmware update service (interval=60s, max_attempts=5, max_elapsed=21600s, attempt_timeout=7200s, boot_debounce=300s)
```

If the values differ from the defaults above, somebody overrode them via env — confirm that's intentional.

### 5d. Browser smoke test

1. Open `https://staging.voltlync.com/admin/firmware`. Confirm:
   - Summary shows 3 cards (Pending / Installed Today / Failed Today) — NOT the old 5-card layout.
   - "How firmware updates work" info card describes BootNotification + exponential backoff.
2. Upload a tiny test firmware (any `.bin` file ≤100 MB).
   - Confirm S3 PUT: `aws s3 ls s3://voltlync-firmware-staging/firmware/ --profile voltlync` shows the new key.
   - Confirm DB row: the new `FirmwareFile` row has `s3_key` populated (not null).
3. Schedule an update against a non-critical staging charger.
   - Active Updates table shows the row with `Attempts: 0`, `Last Attempt: —`, `Next Retry: —`.
   - Within 60s, the scheduler fires UpdateFirmware (if charger is online + idle).
   - `Attempts: 1`, `Last Attempt: <a moment ago>` appears.
4. If the charger is offline, confirm `Attempts: 0` stays — the scheduler waits, doesn't burn attempts on an offline target.

### 5e. Polling-charger auto-detect

If you have a polling (out-of-network) charger or can simulate one:

```bash
curl "https://staging.voltlync.com/api/firmware/latest?external_charger_id=<EXT_ID>&current_firmware_version=<TARGET_VERSION>"
```

If the reported version matches a PENDING update's target, the server returns `null` (auto-closes the row as INSTALLED). Verify the row flipped:

```bash
docker exec ocpp-postgres-staging psql -U ocpp_staging -d ocpp_staging_db -c \
  "SELECT id, status, completed_at FROM firmware_update WHERE charger_id=(SELECT id FROM charger WHERE external_charger_id='<EXT_ID>') ORDER BY id DESC LIMIT 3;"
```

### 5f. Verify FirmwareStatusNotification is logging-only

If any charger sends a `FirmwareStatusNotification` during the smoke test:

```bash
docker logs ocpp-backend-staging 2>&1 | grep "FirmwareStatusNotification from" | tail -5
```

Expected line shape: `📦 FirmwareStatusNotification from <CP>: status=... (informational)`. The row's status in DB should **not** change in response — only BootNotification (with matching version) drives state.

## 6. Rollback procedures

### Decision tree

| Symptom | Path | Time |
|---|---|---|
| Env-var missing (S3 bucket empty / uploads 500) | **Forward-fix** (6a) | ~1 min |
| New code crashes, but `make staging-migrate` has NOT run yet | **Code-only** (6b) | ~5 min |
| Anything went wrong **after** `make staging-migrate` | **Migration downgrade** (6c) or **Full restore** (6d) | ~2 min / ~5 min |
| Just want pre-deploy state exactly | **Full restore** (6d) | ~5 min |

Migration 35's downgrade SQL is clean (it's an Aerich-generated migration, not hand-written) — 6c is usually enough. Full restore (6d) is only needed if data was meaningfully changed post-deploy (e.g. real firmware uploads landed in S3 and you don't want orphan keys).

### 6a. Forward-fix (env / config)

Most common case. Migration succeeded, app is running, but something's wrong with config (typically S3 bucket name).

```bash
# On EC2
cd /home/ec2-user/ocpp-server
nano .env.staging                          # fix the value
make staging-restart                       # `up -d` re-reads --env-file
```

### 6b. Code-only rollback (DB is fine)

Only safe **before** migration 35 has been applied. After migration 35, the old code references `retry_count` which no longer exists.

```bash
# From your laptop
git push origin <rollback-commit-hash>:develop --force

# SSM in
make staging-ssm
cd /home/ec2-user/ocpp-server
make staging-deploy                        # rebuild with old code
```

### 6c. Migration downgrade (DB at v2 → DB at v1, keep new code if you like)

Aerich-generated downgrade SQL is reversible. **However**, any new rows that genuinely use the new state machine (`attempt_count > 0`, `last_attempt_at` set, etc.) will lose those fields. If you've only just deployed, this is fine.

```bash
# On EC2
cd /home/ec2-user/ocpp-server
echo "y" | docker exec -i ocpp-backend-staging aerich downgrade
# Confirms via interactive prompt → answers 'y'
```

After this:
- `firmware_update.attempt_count` → renamed back to `retry_count`
- `firmware_update.last_attempt_at`, `next_retry_at` → dropped
- `firmware_file.s3_key` → dropped (binary remains in S3 if any upload happened)
- The `status` column TYPE changes from `VARCHAR(9)` back to `VARCHAR(19)`. Values stay (`FAILED`/`PENDING`/`INSTALLED`/`CANCELLED` all fit either way).

You can then `git push origin <rollback-hash>:develop --force` and `make staging-deploy` to bring the old code back. Or stay on v2 code with v1 DB (which will crash on startup) — don't do that.

### 6d. Full restore (code + DB from backup)

The reliable option. Same procedure as the GST deploy.

```bash
# On the EC2
cd /home/ec2-user/ocpp-server

# 1. Restore the DB from the newest dump
make staging-restore-db

# 2. Revert code (from your laptop)
git push origin <rollback-commit-hash>:develop --force

# 3. Back on the EC2 — rebuild + restart with the old code
make staging-deploy
```

### What you cannot roll back

- **Orphan S3 firmware blobs.** Any firmware uploaded under v2 stays in `voltlync-firmware-staging` after a DB rollback. Pre-deploy DB has `s3_key=NULL` on every row so nothing breaks. Cleanup is optional: `aws s3 rm s3://voltlync-firmware-staging/firmware/ --recursive --profile voltlync`.
- **In-flight firmware downloads.** If a charger is mid-download when you roll back, it'll finish its download from the (still-valid) presigned URL and reboot with the new firmware. The DB will then think the charger is on the old version (because the row doesn't exist anymore). Resolve by waiting for the next BootNotification, which will update `charger.firmware_version`.

## 9. Known follow-ups (not blocking)

- **Historical local-file migration to S3.** Existing `FirmwareFile` rows have `s3_key=NULL` and serve via the legacy `/firmware/<filename>` StaticFiles mount. A one-time script (`backend/scripts/migrate_firmware_to_s3.py`, not in this PR) will backfill `s3_key` for those rows. Once done, the `/firmware` mount in `main.py` can be removed and the `backend_firmware` Docker volume retired. Plan/run as a separate maintenance task.
- **Drift regression test.** A pytest fixture that asserts `aerich.content`'s latest row matches current `models.py`. Would guard against the snapshot-poisoning issue described in `MEMORY.md` → `feedback_stop_before_handwriting_migration.md`. Not blocking.
- **WS-drop-during-firmware-update alarm suppression.** The new code marks `expected_ws_drop_until` in `connected_charge_points` when UpdateFirmware is sent successfully. If Sentry/New Relic alerts are noisy during a firmware deploy, follow up by wiring the disconnect handlers to honor this flag.
- **OCPP `retries=3, retry_interval=300s` in the UpdateFirmware payload.** Kept as-is — charger-internal retries are still useful for transient sub-minute blips. Revisit if vendor docs suggest tuning.
